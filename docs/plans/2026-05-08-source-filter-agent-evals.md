# Source-Filtered Agent Evals Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Make `npm run network:evaluate` catch regressions in source-specific Hermes/MCP workflows: source filters must fail closed, source-backed answers must include matched source evidence, and stale/empty source questions must use `source_health` instead of vibes.

**Architecture:** Extend the existing deterministic eval harness rather than adding another tool. Keep `crm/evaluation.js` as the pure contract checker, teach `scripts/evaluate-network-memory.js` to run `query_network` and selected MCP tool cases from `tests/fixtures/agent-workflows.json`, and add synthetic source-filter fixtures generated by `scripts/seed-dev-data.js`. The eval suite remains local, read-only, non-LLM, and synthetic-only.

**Tech Stack:** Plain Node.js CommonJS, Node built-in test runner, existing `crm/evaluation.js`, `crm/agent-retrieval.js`, `crm/agent-source-health.js`, `scripts/evaluate-network-memory.js`, `scripts/minty-mcp-server.js`, `scripts/seed-dev-data.js`, `tests/fixtures/agent-workflows.json`, `tests/unit/evaluation.test.js`, `tests/unit/seed-dev-data.test.js`, `package.json`.

---

## Product framing

Minty's weekly reliability theme is source quality and network-memory trust. The repo now has `source_health`, source filters on `search_network`, refresh diagnostics, and a small workflow eval harness. The gap is that the eval harness currently exercises only three `query_network` cases and does not prove the source-specific contract that Hermes will actually depend on.

That is dangerous because source-filter bugs are subtle: a Telegram query can accidentally return LinkedIn profile matches; an invalid source can silently degrade to all-source search; a stale source can still produce confident prose; and MCP envelopes can drift from the CLI contract. This plan turns those product promises into deterministic synthetic gates.

This complements, not duplicates:

- `2026-05-06-agent-source-health-mcp.md`: implemented the source readiness preflight surface.
- `2026-05-06-agent-workflow-evals.md`: established the first eval harness; this plan narrows the next iteration to source-filter and MCP parity.
- `2026-05-07-memory-refresh-diagnostics.md`: reports pipeline freshness; these evals prove the answering layer respects source trust.
- Current `main` already exposes exactly four MCP tools in `tests/unit/minty-mcp-server.test.js`: `person_context`, `search_network`, `source_health`, and `workflow_brief`. This plan should not change that exact tool-list assertion; it routes evals through the existing `source_health` tool.
- Current `scripts/evaluate-network-memory.js` still filters default cases down to `query_network`, and `crm/evaluation.js` still passes only `testCase.query` into the runner. Those are the narrow implementation gaps this plan closes.

## Success criteria

- `npm run network:evaluate` runs all cases in `tests/fixtures/agent-workflows.json`, not only bare `query_network` cases.
- Eval cases can pass tool arguments like `{ query, source }` or `{ source }`, not only a query string.
- A Telegram-filtered synthetic query returns only results with `matchedSources: ["telegram"]` and evidence from Telegram/contact evidence.
- A LinkedIn-filtered version of the same Telegram-only query returns zero results and does not fallback to all-source ranking.
- An invalid source filter returns zero results and diagnostics/safety metadata that make the failure explicit with sanitized sentinel fields, without echoing unsafe input.
- `mcp:source_health` evals assert source readiness without returning people or contact details.
- The fixtures remain synthetic-only: no real emails, phones, raw messages, private paths, token names, Sree data, or private strategy notes.

## Non-goals

- Do not change ranking semantics except where tests expose a source-filter contract bug.
- Do not add an LLM judge, embeddings, network calls, new npm dependencies, or real data fixtures.
- Do not add a new MCP tool, UI view, outreach action, or refresh command.
- Do not weaken existing evals to make new source-filter cases pass; fix seed data or implementation instead.
- Do not expose raw message bodies, raw contact ids, source IDs, group names, emails, phones, token paths, or private brain paths.

Add this synchronous helper below `readCases()`:

```js
function runAgentEvalCase(testCase, data) {
    const target = testCase && testCase.target;
    const args = (testCase && testCase.arguments) || {};
    if (target === 'query_network') {
        const query = args.query || testCase.query || '';
        const opts = { ...data, limit: 10 };
        if (args.source) opts.source = args.source;
        if (args.sources) opts.sources = args.sources;
        return queryNetwork(query, opts);
    }
    if (typeof target === 'string' && target.startsWith('mcp:')) {
        const tool = target.slice('mcp:'.length);
        const resp = handleMessage({
            jsonrpc: '2.0',
            id: 1,
            method: 'tools/call',
            params: { name: tool, arguments: args },
        }, data);
        const text = resp && resp.result && resp.result.content && resp.result.content[0] && resp.result.content[0].text;
        return JSON.parse(text || '{}');
    }
    throw new Error(`Unsupported eval target: ${target}`);
}
```

Update `main()` to route each case through the helper:

```js
function main() {
    const args = parseArgs(process.argv.slice(2));
    const data = loadData(args.dataDir);
    const cases = readCases(args.casesPath);
    const report = evaluateRelationshipQueries(cases, (testCase) => runAgentEvalCase(testCase, data));
    console.log(JSON.stringify({ dataDir: args.dataDir, ...report }, null, 2));
    process.exitCode = report.failed ? 1 : 0;
}
```

`handleMessage()` is synchronous, so keep the eval runner synchronous. Tests may still use `await runAgentEvalCase(...)`; awaiting a non-Promise value is valid and keeps future async refactors easy.

Export the helper:

```js
module.exports = { DEFAULT_CASES, parseArgs, runAgentEvalCase };
```
